# Task Service

Сервис для управления задачами с HTTP API на Go.

## Быстрый запуск

docker compose up --build

После запуска сервис будет доступен по адресу http://localhost:8080

Swagger UI: http://localhost:8080/swagger/

---

## Реализация периодических задач

### Анализ требования

В существующей системе была таблица tasks без поддержки повторяющихся задач. Требовалось добавить функциональность для:
- Ежедневных задач с интервалом N дней
- Ежемесячных задач в указанные дни месяца
- Задач на конкретные даты
- Задач по четным/нечетным дням месяца

### Принятые архитектурные решения

#### 1. Отделение шаблонов от экземпляров задач

Создана таблица task_templates для хранения правил повторения, а фактические задачи остаются в таблице tasks. Связь реализована через task_instances.

Обоснование: Это позволяет изменять правила, не затрагивая уже созданные задачи, отменять конкретные экземпляры без удаления шаблона и хранить историю генерации.

#### 2. Транзакционное создание

При создании задачи с периодичностью все операции выполняются в одной транзакции: вставка шаблона в task_templates, вставка задачи в tasks, создание связи в task_instances.

Обоснование: Гарантирует целостность данных — либо все операции успешны, либо полностью откатываются.

#### 3. Умная логика для ежемесячных задач

Если указанный день месяца (например, 31) отсутствует в текущем месяце, задача создаётся в последний доступный день.

Обоснование: Предотвращает пропуск задач и соответствует ожиданиям пользователя.

#### 4. JSONB для гибких данных

Поля monthly_days и specific_dates хранятся в формате JSONB.

Обоснование: Позволяет хранить массивы переменной длины без изменения схемы БД.

#### 5. Автоматическая генерация через Worker

Фоновый воркер проверяет каждый час активные шаблоны и создаёт задачи для подходящих дат.

Обоснование: Разгрузка API, единый источник создания повторяющихся задач.

### Схема БД

Таблица task_templates:
- id (PK)
- title
- description
- recurrence_type
- daily_interval
- monthly_days (JSONB)
- specific_dates (JSONB)
- parity_type
- start_date
- end_date
- execution_time
- status
- created_at
- updated_at

Таблица tasks:
- id (PK)
- title
- description
- status
- created_at
- updated_at

Таблица task_instances:
- id (PK)
- template_id (FK -> task_templates.id)
- task_id (FK -> tasks.id)
- due_date
- generated_at
- is_cancelled

### Граничные случаи

Сценарий: 31 февраля → Решение: Автоматически заменяется на 28/29
Сценарий: start_date в будущем → Решение: Задачи не создаются до этой даты
Сценарий: end_date истёк → Решение: Шаблон игнорируется
Сценарий: Интервал 0 или отрицательный → Решение: Валидация, запись не создаётся
Сценарий: Нет активных шаблонов → Решение: Worker ничего не генерирует
Сценарий: Дублирование (шаблон + дата) → Решение: ON CONFLICT в task_instances

### API эндпоинты

POST /api/v1/tasks - Создание задачи (с опциональной периодичностью)
GET /api/v1/tasks - Список всех задач
GET /api/v1/tasks/{id} - Получение задачи по ID
PUT /api/v1/tasks/{id} - Обновление задачи
DELETE /api/v1/tasks/{id} - Удаление задачи

### Пример создания периодической задачи

Запрос:
{
  "title": "Ежемесячный отчёт",
  "description": "Сбор данных за месяц",
  "status": "new",
  "recurrence": {
    "recurrence_type": "monthly",
    "monthly_days": [15],
    "start_date": "2026-05-01T00:00:00Z"
  }
}

Ответ:
{
  "id": 1,
  "title": "Ежемесячный отчёт",
  "description": "Сбор данных за месяц",
  "status": "new",
  "created_at": "2026-05-01T12:34:56Z",
  "updated_at": "2026-05-01T12:34:56Z"
}

### Технологии

- pgx/v5

- gorilla/mux
- Swagger

### Допущения и ограничения

1. Время выполнения: задачи создаются с датой, время всегда 00:00:00
2. Worker интервал: проверка каждый час
3. Часовые пояса: все даты хранятся в UTC
4. Обновление шаблонов: после создания не редактируются
5. Конфликты: повторная генерация предотвращается

### Итог

Реализована поддержка периодических задач с чистой архитектурой, транзакционной целостностью и автоматической генерацией через фоновый воркер. Код готов к расширению и эксплуатации.
